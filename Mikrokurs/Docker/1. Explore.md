# _Let's Explore_ Docker
Witajcie w części pierwszej modułu `Docker`! Czeka was szybkie zapoznanie się z tym jak ściągnąć, zainstalować i użyć Dockera. Zaczniemy od uruchomienia maszyny wirtualnej, żeby nie namieszać sobie za bardzo w systemie.

## 1. Maszyna Wirtualna
1. Najpierw ściągnij [VirtualBox](https://www.virtualbox.org/), jeśli jeszcze go nie masz.
2. Następnie ściągnij [obraz ZSK](https://lindev.cs.put.poznan.pl/ZSK/ZSK.ova) – jest to obraz dystrybucji OpenSUSE, takiej samej jak używana na zajęciach – i stwórz maszynę na podstawie pobranego obrazu.

Po uruchomieniu maszyny logujesz się jako `student`. Uprawnienia roota możesz uzyskać wykonując na przykład:
```console
$ sudo -i
# # <-- zobaczysz czerwony prompt z symbolem '#' – to znaczy, że jesteś rootem w konsoli ale także W TYM PLIKU
```

Uruchom usługę systemową Docker, która będzie działać w tle dając Ci dostęp do wszystkich funkcji dockera – ona odpowiada za działanie docker engine.

```console

```


> [!TIP]
> Żeby korzystać z Dockera jako student możecie przeprowadzić następujące kroki:
> 
> 5. Create the `docker` group.
> ```console
> # groupadd docker
> ```
> 
> 6. Add your user to the `docker` group.
> ```console
> $ sudo usermod -aG docker $USER
> ```
>
> 7. Log out and log back in so that your group membership is re-evaluated. (If you're running Linux in a virtual machine, it may be necessary to restart the virtual machine for changes to take effect.)
>
> You can also run the following command to activate the changes to groups:
> ```console
> $ newgrp docker
> ```
> You can have a look at the documentation for the full, official [post-install setup](https://docs.docker.com/engine/install/linux-postinstall/).

# 3. _Playground_

Poszukaj w rejestrze obrazów Dockerowych `ubuntu`:
```console
$ docker search ubuntu
```

Następnie pobierz go na swoją maszynę i zweryfikuj jakie obrazy masz już pobrane:
```console
$ docker pull ubuntu
$ docker images
```

Teraz możemy wystartować ubuntu skonteneryzowane na naszej maszynie, np.:
```console
$ docker run --name ubuntuc -i –t ubuntu /bin/bash
```
Po uruchomieniu go powinniście być w bashu na ubuntu.

W osobnym terminalu możecie sprawdzić jakie procesy Docker aktualnie utrzymuje:
```console
$ docker ps -a
```

Teraz możecie zatrzymać kontener używając jego nazwy lub hasha (który dostaliście w odpowiedzi na z `docker run`):
```console
$ docker stop 72c9629
```
albo:
```console
$ docker stop cubuntuc
```

Taki zatrzymany kontener można oczywiście przywrócić do życia:
```console
$ docker start ubuntuc
$ docker attach ubuntuc
```
albo za jednym razem:
```console
$ docker start --attach ubuntuc
```

## Bardziej złożony przykład
Pobawiliśmy się ubuntu, teraz pora na coś bardziej przydatnego: PostgreSQL w Dockerze.

> [!WARNING]
> Zacznij od pobrania obrazu `postgres` tak samo jak pobrałeś wcześniej `ubuntu`.

Tym razem uruchamiając kontener wykorzystamy jeszcze jedną istotną rzecz – ustawimy _hostname_, żebyśmy byli pewni, że pracujemy na bazie danych: 
```console
$ docker run -h database --name postgres_db -e POSTGRES_PASSWORD=secret postgres
```

Zobaczysz masę logów. Są to oczywiście logi uruchomionego w kontenerze serwera postgresa. Jak w takim razie wejść w interakcję z tym kontenerem? Możemy np. wykonać w osobnym terminalu polecenie `bash` w naszym postgresie, niezależnie od działającego w tym samym czasie kontenera.
```console
$ docker exec -it postgres_db /bin/bash

root@database:/#
```

Będąc w środku możesz zalogować się do bazy danych używając polecenia `psql`, na przykład:
```console
root@database:/# psql -U postgres
```
albo:
```console
root@database:/# su - postgres
postgres@database:~$ psql

psql (18.3 (Debian 18.3-1.pgdg13+1))

Type "help" for help.

  

postgres=#
```

Jeśli znasz Postgresa możesz się pobawić dalej tym kontenerem, ale na razie skupmy się jeszcze chwilę na dockerze.

Zatrzymaj `postgres_db` (i upewnij się, że `ubuntuc` też jest zatrzymany). Możesz usunąć zatrzymane kontenery używając:
```console
$ docker rm postgres_db
```

> [!WARNING]
> Tak samo usuń `ubuntuc`!

Odpalając kontener możesz od razu zażyczyć sobie, by został usunięty jak tylko skończy działanie:
```console
$ docker run --rm ...
```

### Docker Compose
Oczywiście wszystkie kontenery dockerowe można stawiać ręcznie, tak jak robiliśmy to wcześniej, ale staje się to żmudne gdy mamy do postawienia wiele kontenerów i/lub wiele konfiguracji do podania. Teraz zerkniemy jak to zrobić szybciej.

Stwórzcie nowy katalog `docker` a w nim plik, nazwany `docker-compose.yaml`, z następującą zawartością:
```yaml
services:  
  db:  
    image: postgres  
    environment:  
      POSTGRES_USER: postgres  
      POSTGRES_PASSWORD: postgres  
      POSTGRES_DB: postgres  
    ports:  
      - "5432:5432"  
    healthcheck:  
      test: ["CMD-SHELL", "pg_isready -U postgres"]  
      interval: 3s  
      timeout: 3s  
      retries: 10  
  
  server:  
    image: ghcr.io/konrad-szychowiak/edu:node-1  
    ports:  
      - "3000:3000"  
    environment:  
      SIGNING_KEY: your-super-secret-key  
      POSTGRES_HOST: db  
      POSTGRES_PORT: 5432  
      POSTGRES_USER: postgres  
      POSTGRES_PASSWORD: postgres  
      POSTGRES_DB: postgres  
    depends_on:  
      db:  
        condition: service_healthy
```

Może już widzicie, że przygotowaliśmy sobie dwie usługi – bazę danych i serwer. Uruchomimy je teraz (w tym katalogu, w którym jest nasz plik):
```console
$ docker-compose up
[+] up 2/2
 ✔ Container docker-db-1  Healthy      0.6s
 ✔ Container docker-server-1 Started   0.7s
```

Serwer który uruchomiliśmy jest bardzo prosty: możemy stać się jednym z użytkowników, zobaczyć czy jesteśmy obecnie jakimś użytkownikiem i jeśli tak, to odczytać z bazy jakie pliki mamy zapisane. Do dyspozycji mamy następujące zapytania:
- `GET /become/{user_id}` – zaloguj się jako `user_id`
- `GET /whoami` – potwierdź czy jesteś zalogowany
- `GET /files` – odczytaj pliki dla użytkownika, jako który jesteś zalogowany

Zatem musimy znać id użytkownika, żeby się zalogować. Problem? Żaden endpoint nie listuje id. Może jednak jesteśmy od razu zalogowani? Sprawdźmy z poziomu konsoli:
```console
$ curl -XGET -H "Content-type: application/json" 'localhost:3000/whoami'

{"error":"Missing or invalid Authorization header"}
```

Wygląda na to, że nie jesteśmy. Jak sobie z tym poradzić? Na całe szczęście widzieliśmy już jak połączyć się z bazą danych:
```console
$ docker exec -it docker-db-1 /bin/bash
root@fe6119ea87fa:/# psql -U postgres
```

> [!TIP]
> W celu pisania mniej poleceń możemy od razu połączyć się z postgresem
> ```console
> $ docker exec -it docker-db-1 psql -U postgres
> psql (18.3 (Debian 18.3-1.pgdg13+1))
> Type "help" for help.
> postgres=#
> ```

Możemy wypisać tabele z bazy danych:
```console
postgres=# \d

              List of relations

 Schema |     Name     |   Type   |  Owner   

--------+--------------+----------+----------

 public | files        | table    | postgres

 public | files_id_seq | sequence | postgres

 public | users        | table    | postgres

(3 rows)
```

I wybrać dane z tabeli users:
```console
postgres=# SELECT * FROM users;

                  id                  |  name   

--------------------------------------+---------

 de17d901-42b7-430d-b958-717282fc6617 | root

 5583b60e-7c5b-48e0-a653-6308745d3caa | student

 adb8a2c3-6bfd-4b38-bce9-627c4cff5579 | newbie

(3 rows)
```

> [!WARNING]
> ID użytkowników są generowane losowo, więc wasze będą inne.

Możemy teraz już z poziomu konsoli zalogować się jako jeden z użytkowników. Zaczniemy od roota:
```console
$ curl -XGET -H "Content-type: application/json" 'localhost:3000/become/de17d901-42b7-430d-b958-717282fc6617'

{"token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6ImRlMTdkOTAxLTQyYjctNDMwZC1iOTU4LTcxNzI4MmZjNjYxNyIsIm5hbWUiOiJyb290IiwiaWF0IjoxNzc0MTAwMjczfQ.dCp6aQWqFmuYjrupFoUXdAXCBanb4huzmy32uQX3a-4"}
```

Otrzymaliśmy token! Wykorzystajmy go, żeby sprawdzić czy jesteśmy zalogowani:
```console
$ curl -XGET -H 'Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6ImRlMTdkOTAxLTQyYjctNDMwZC1iOTU4LTcxNzI4MmZjNjYxNyIsIm5hbWUiOiJyb290IiwiaWF0IjoxNzc0MTAwMjczfQ.dCp6aQWqFmuYjrupFoUXdAXCBanb4huzmy32uQX3a-4' -H "Content-type: application/json" 'localhost:3000/whoami'
{"id":"de17d901-42b7-430d-b958-717282fc6617","name":"root"}
```

> [!WARNING]
> Super! To teraz, już sami spróbujcie sprawdzić jakie pliki mają wszyscy użytkownicy!
